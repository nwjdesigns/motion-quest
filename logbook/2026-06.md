## 2026-06-16 (s2) — On-scene control panel + custom colour picker

**Tool**: Cavalry WASM web player + Astro site  ·  **Tags**: control-centre, wasm-api, color-picker, TDD, scene-interactivity

### What I made
Real scene interactivity: an on-scene **control panel** that reads whatever Control Centre attributes a `.cv` exposes and renders them as live widgets (top-right, frosted blur, over the player), plus a **custom HSV colour picker** that replaces the un-stylable native OS picker. Also shrank the panel 10% and relabelled the detail-page nav.

### How
- **Discovery first.** Instrumented `player.html` to dump scene 01's real Control Centre schema. Finding: scene 01 only exposes **two** attributes — `compNode#134.backgroundColor` (color) and `stagger#15.strength` (double, unbounded). The five controls in Noah's reference screenshot (Shape/Width/Color/Radius/Inverted) came from a DIFFERENT Cavalry demo scene — they were the visual reference, not 01's controls. So the panel had to be dynamic, not hardcoded.
- **Cavalry web-player API (mapped this session):** `player.getActiveComp()` → `getControlCentreAttributes(compId)` (vector of `layerId.attrId`), `getAttributeName`, `getAttributeDefinition` (`.type`, `.numericInfo` min/max/step, `.enumValues`, `.multiline`), `getAttribute`/`setAttribute`. **The typed setters/getters are broken** in this WASM build — `setAttributeColor`/`getAttributeColor` return `{0,0,0,0}`. Use the GENERIC `setAttribute`/`getAttribute` with `{r,g,b,a}` (0-255). No group/layer-name API, so the demo's "PATH" header can't be reproduced.
- **TDD seam.** `public/cavalry/control-centre.js` (pure: `describeControl`, `colorToHex`/`hexToColor`, `rgbToHsv`/`hsvToRgb`, `resolveRange`, `computeFillPct`, `formatNumber`, `shouldRender`) with `tests/control-centre.test.ts` (31 tests). `public/cavalry/render-controls.js` owns the DOM + WASM writes (untestable layer kept thin). Wired into `player.html` (CSS + a `renderControls` call). 172 → **203 tests**.
- **Custom colour picker:** SV gradient square + rainbow hue strip + editable hex, in a frosted rounded popover anchored under the colour row, replacing `<input type=color>`. Works in HSV, converts to rgb only on write. Hardened `attachDrag` so a plain click updates and a `setPointerCapture` throw can't swallow the first input.
- **Polish:** whole panel `transform: scale(0.9)` (anchored top-right); picker padding 12→8; detail-page nav "← Back" → **"Gallery"** (no arrow) so only Prev/Next carry arrows (Noah picked this over up-arrow / grid-icon options).

### What clicked / gotchas
- Typed Cavalry WASM attribute setters are unreliable; generic `setAttribute` is the safe path (and is what the `CavalryPlayer.js` spike used all along).
- `preview_screenshot` actually captured the HTML control panel + picker fine (it's DOM over the canvas) — the known WASM-iframe glitch is only the canvas paint.
- Verifying interactivity required driving real pointer events on the widgets and reading WASM values back, not just trusting DOM updates (that's how the broken `setAttributeColor` was caught — swatch updated but the scene value went to zero).

### Decisions locked
- Scene controls live **docked top-right inside the player iframe** (player-rendered), NOT merged into the left card (an earlier answer that a later screenshot superseded). No host↔iframe bridge needed.
- Custom picker over native OS picker (native is un-stylable).
- Nav: "Gallery" instead of "← Back".

### Output
- `public/cavalry/control-centre.js`, `public/cavalry/render-controls.js`, `tests/control-centre.test.ts` (new)
- `public/cavalry/player.html` (control panel CSS + wiring), `src/pages/experiments/[...slug].astro` (Gallery)

This is the resumed P1 (info-panel type hierarchy).

## 2026-06-23 — Remove morph transition system

**Tool**: Astro/React site  ·  **Tags**: transition, bugfix, simplification

### What I made
Removed the entire seamless morph transition system (forward morph, reverse morph, chrome-exit animation). Clicking a gallery node now navigates directly to the detail page. Camera state persistence still works.

### How
- **Root-cause diagnosis.** Forward morph was completely broken: clicking a node faded the chrome out but never navigated. The bug: `useMarkBehaviour` calls `setState` on every animation frame (~60fps), re-rendering the parent. Each re-render created a new `onChromeExit` arrow function. Since `onChromeExit` was in MorphSnapshot's `useEffect` dependency array, the WAAPI morph animation was cancelled and restarted every frame, never completing.
- **Noah's call:** even after fixing the identity bug (wrapping in `useCallback`), there was still a visible glitch on click. Noah had anticipated this and decided to can the seamless transition idea entirely.
- **Removal (4 files, -268 lines, +6 lines):**
  - Deleted `MorphSnapshot.tsx`
  - `ConstellationScene.tsx`: removed forward morph state (`morphState`, `chromeExiting`), reverse morph state + `ReverseMorphOverlay` component + `readReverseMorphData`, chrome-exit style logic, `handleNavigate` callback, MorphSnapshot/morph-animation/transition-orchestrator imports
  - `ExperimentNode.tsx`: removed `onNavigate` prop, projection logic (`Matrix4`, `projectRect`), direct navigate on click
  - `[...slug].astro`: removed reverse morph script (Gallery link intercept, popstate handler, `capturePlayerRect`, `storeReverseMorph`)
- 356 tests still pass. Library files (`morph.ts`, `morph-animation.ts`, `transition-orchestrator.ts`, `reverse-morph.ts`) and their tests left in place as dead code.

### Decisions locked
- **No seamless transition.** Direct navigation between gallery and detail page. The morph concept is canned.

### Output
- `src/components/ConstellationScene.tsx`, `src/components/ExperimentNode.tsx`, `src/pages/experiments/[...slug].astro` (modified)
- `src/components/MorphSnapshot.tsx` (deleted)
